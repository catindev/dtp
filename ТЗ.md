# Техническое задание на прототип: Don’t Touch Prod — Kanban Pivot

> **Архивный документ.** Это ТЗ описывает ранний пошаговый прототип с большим количеством колонок, кнопочным управлением, fatigue/morale и ручным релизом. Актуальное состояние проекта описано в `Диздок.md`; хронология изменений - в `CHANGELOG.md`.

**Версия:** 0.1
**Цель:** разработать минимальный игровой прототип для проверки ядра
**Формат:** локальный playable prototype
**Приоритет:** системная проверка, не полировка
**Ключевая гипотеза:** управление процессом, командой и риском релиза даёт реиграбельный выбор

---

## 1. Назначение прототипа

Прототип должен проверить, есть ли игра в следующем цикле:

```txt
карточки приходят →
игрок ведёт их по канбан-доске →
назначает персонажей →
персонажи прорабатывают карточки и устают →
игрок сам решает, когда релизить →
релиз вскрывает последствия →
игрок адаптирует процесс и команду
```

Прототип не должен доказывать визуальный стиль, narrative, реализм разработки или полноту контента. Он должен ответить на вопрос:

> Возникают ли у игрока регулярные ситуации с 3+ разумными решениями, каждое из которых имеет цену?

## 3. Архитектурное требование

Игровая логика должна быть отделена от UI. Желательно использовать Redux паттерн.

Обязательное разделение:

```txt
core simulation
presentation/UI
content data
```

### 3.1. Core simulation

Содержит:

- GameState;
- TaskCard;
- Character;
- генератор задач;
- генератор персонажей;
- work resolver;
- release resolver;
- day resolver;
- sprint resolver.

Core не должен зависеть от сцен Godot.

### 3.2. Presentation/UI

Содержит:

- канбан-доску;
- карточки;
- панель персонажей;
- панель ресурсов;
- лог событий;
- кнопки действий.

UI вызывает методы core и отображает состояние.

### 3.3. Content data

Контент хранить отдельно от логики.

Минимально допустимо:

```txt

```

или Godot Resource, если быстрее.

Предпочтительно для прототипа:


## 5. Минимальный игровой сценарий

Игрок начинает забег.

Старт:

```txt
sprint: 1
day: 1
daysPerSprint: 5
maxSprints: 4

trust: 50
debt: 20
value: 0
budget: 3

wipLimit: 4
```

Стартовая команда:

```txt
1 analyst
1 designer
1 backend
1 frontend
```

Стартовый backlog:

```txt
3 карточки:
- feature
- bug
- techDebt
```

Каждый день игрок:

1. видит карточки на доске;
2. перемещает карточки между колонками;
3. назначает персонажей на работу или отдых;
4. нажимает End Day;
5. смотрит результаты;
6. при желании релизит карточки.

Каждый спринт:

1. длится 5 дней;
2. в конце показывается review;
3. игрок получает budget;
4. игрок может нанять или обучить персонажа;
5. начинается следующий спринт.

Забег длится 4 спринта.

## 7. Work tracks

Список work tracks фиксированный:

```txt

```

Назначение:

| Track         | Что делает                            |
| ------------- | ------------------------------------- |
| analysis      | снижает риск wrong scope / dependency |
| design        | снижает UX/product риск               |
| backend       | делает backend часть                  |
| frontend      | делает frontend часть                 |
| review        | снижает technical/regression риск     |
| qa            | раскрывает баги и регрессии           |

## 11. Work resolver

### 11.1. Требование

Реализовать функцию:

```txt
workOnTask(state, characterId, taskId, workTrack) -> WorkResult
```

---

### 11.2. WorkResult

```txt
WorkResult:
  progressGain: int
  revealedRisks: TaskRisk[]
  mitigatedRisks: TaskRisk[]
  xpGain: int
  fatigueGain: int
  burnoutGain: int
  createdDefect: TaskRisk | null
```

---

### 11.3. Алгоритм

```txt
1. Проверить, что персонаж доступен.
2. Проверить, что карточка не done.
3. Посчитать effectiveSkill.
4. Посчитать progressGain.
5. Добавить progress в task.progress[workTrack].
6. Добавить characterId в task.contributors.
7. Попробовать раскрыть hidden risks.
8. Попробовать mitigated revealed risks.
9. Посчитать шанс создать defect.
10. Начислить XP.
11. Начислить fatigue.
12. Начислить burnout.
13. Обновить focusTaskId.
14. Записать GameEvent.
```

---

### 11.4. Effective skill

```txt
effectiveSkill =
  base skill
  + trait bonuses
  - fatigue penalty
  - burnout penalty
```

Fatigue penalty:

```txt
floor(fatigue / 30)
```

Burnout penalty:

```txt
floor(burnout / 50)
```

Итог:

```txt
clamp(effectiveSkill, 0, 6)
```

---

### 11.5. Progress gain

Базовая формула:

```txt
progressGain = 6 + effectiveSkill * 5
```

Модификаторы:

```txt
fast: x1.2
careful: x0.85
off-role: x0.75
context switch: x0.85
task size: divide by sizeModifier
random: x0.8..1.2
```

Ограничение:

```txt
min 2
max 35
```

---

### 11.6. Раскрытие риска

Для каждого риска:

Условия:

```txt
risk.state == hidden
workTrack in risk.discoverBy
```

Шанс:

```txt
chance =
  0.15
  + effectiveSkill * 0.12
  + careful bonus
  - fatigue * 0.003
  - burnout * 0.002
```

Если успех:

```txt
risk.state = revealed
risk.triggerText добавляется в visible.revealedRisks
```

---

### 11.7. Mitigation риска

Для каждого риска:

Условия:

```txt
risk.state == revealed
workTrack in risk.mitigateBy
task.progress[workTrack] >= task.trueNeeds[workTrack] * 0.75
```

Если условия выполнены:

```txt
risk.state = mitigated
```

---

### 11.8. Создание дефекта

При работе может появиться новый hidden risk.

Шанс дефекта:

```txt
base 5%
+ sloppy 15%
+ fast 10%
+ fatigue over 70: 10%
+ burnout over 60: 10%
- careful 10%
```

Если сработало:

```txt
добавить hidden risk типа backend_bug/frontend_bug/regression
```

В первом прототипе можно включить только для tracks:

```txt
backend
frontend
```

## 12. XP и прокачка

### 12.1. XP gain

```txt
xp = 1
+ floor(progressGain / 15)
+ 2 if revealed risk
+ 3 if mitigated risk

xp *= character.growthRate
```

XP начисляется в track, по которому персонаж работал.

---

### 12.2. Skill level up

Порог:

```txt
threshold = 8 + currentSkill * 6
```

Если XP >= threshold:

```txt
skill +1
xp -= threshold
```

Ограничение:

```txt
skill max 5
```

---

## 13. Усталость и выгорание

### 13.1. Fatigue gain

Базовая работа:

```txt
+12 fatigue
```

Модификаторы:

```txt
task pressure: +pressure
off-role work: +4
context switch: +5
incident/hotfix: +6
overtime: +12
fragile: x1.3
```

В первом прототипе overtime можно не делать.

---

### 13.2. Burnout gain

Burnout растёт, если персонаж работает уставшим.

Формула:

```txt
burnoutGain = 0

if fatigue > 60:
  +2

if fatigue > 80:
  +4

if task.kind == incident:
  +3

if task.pressure >= 4:
  +1

if fragile:
  x1.5

divide by resilience
round
```

---

### 13.3. Passive recovery

В конце дня:

Если персонаж работал:

```txt
fatigue -6
```

Если отдыхал:

```txt
fatigue -35
burnout -4
```

Ограничения:

```txt
fatigue min 0
burnout min 0
```

---

### 13.4. Sick leave

Если burnout >= 100:

```txt
status = sickLeave
unavailableDays = 3
burnout = 75
fatigue = 50
```

Каждый день sick leave:

```txt
unavailableDays -1
fatigue -20
burnout -5
```

Когда unavailableDays == 0:

```txt
status = available
```

```txt

```

---

### 14.2. Формула

```txt

```

Если:

```txt
task.age > task.deadline
```

То:

```txt

```

```txt

```

Для incident:

```txt
trust loss x2
```

## 15. WIP friction

### 15.1. Active WIP

Считать карточки в колонках:

```txt
analysis
design
dev
review
qa
release
```

Incoming не считать.

---

### 15.2. Формула

Если:

```txt
activeWip > wipLimit
```

То:

```txt
overflow = activeWip - wipLimit

for each available character:
  fatigue += overflow * 2

debt += overflow
```

Цель: открытая работа создаёт шум и утомляет команду.

## 16. Release resolver

### 16.1. Требование

Реализовать функцию:

```txt
releaseTask(state, taskId) -> ReleaseOutcome
```

Карточку можно релизить из любой активной колонки. UI может предлагать кнопку Release на выбранной карточке.

---

### 16.2. Release risk

Считать riskScore:

```txt
riskScore = 0
```

Для каждого workTrack:

```txt
gap = max(0, trueNeeds[track] - progress[track])
riskScore += gap * trackRiskWeight[track]
```

Weights:

```txt
analysis: 0.12
design: 0.08
backend: 0.16
frontend: 0.12
review: 0.10
qa: 0.18
releaseSafety: 0.14
```

Риски:

```txt
hidden risk:
  riskScore += severity * 8

revealed risk:
  riskScore += severity * 4

mitigated risk:
  riskScore += severity * 1
```

Глобальное состояние:

```txt
riskScore += debt * 0.25
```

Усталые contributors:

```txt
for each contributor:
  if fatigue > 70 or burnout > 60:
    riskScore += 5
```

Release safety:

```txt
riskScore -= progress.releaseSafety * 0.3
```

Итог:

```txt
riskScore = clamp(riskScore, 0, 100)
```

---

### 16.3. Outcome tiers

```txt
0..20:
  clean release

21..45:
  rough success

46..70:
  bug escaped

71..100:
  incident
```

Можно сделать deterministic outcome по score для первого прототипа. Рандом внутри outcome не обязателен.

---

### 16.4. Effects

Clean release:

```txt
value += task.value
trust += 4
debt += 0
```

Rough success:

```txt
value += task.value
trust += 1
debt += 2
30% chance spawn follow-up bug
```

Bug escaped:

```txt
value += round(task.value * 0.7)
trust -= 4
debt += 5
spawn bug card
```

Incident:

```txt
value += round(task.value * 0.3)
trust -= 12
debt += 8
spawn incident card
contributors burnout += 5
```

TechDebt card special:

Если techDebt релизится clean/rough:

```txt
debt -= task.value
value += round(task.value * 0.3)
```

```txt

```

Модификаторы:


Ограничение:

## 22. Acceptance criteria

Прототип считается готовым, если выполняются условия ниже.

### 22.1. Игровой цикл

Игрок может:

- начать новый run;
- видеть ресурсы;
- видеть доску;
- видеть карточки;
- видеть персонажей;
- перемещать карточки;
- назначать персонажей;
- отправлять персонажей отдыхать;
- завершать день;
- релизить карточки;
- получать последствия релиза;
- дойти до sprint review;
- завершить run победой или поражением.

---

### 22.2. Системы карточек

Карточки:

- генерируются;
- имеют тип, домен, pressure, deadline, value, size;
- имеют hidden risks;
- имеют trueNeeds;
- имеют visible knownNeeds;
- имеют progress по tracks;
- стареют;
- могут создавать follow-up.

---

### 22.3. Системы персонажей

Персонажи:

- имеют роль;
- имеют seniority;
- имеют skills;
- имеют traits;
- получают fatigue;
- получают burnout;
- получают XP;
- прокачивают skills;
- могут уходить на sick leave.

---

### 22.4. Work resolver

При работе:

- progress карточки растёт;
- risk может раскрыться;
- risk может mitigated;
- персонаж получает XP;
- персонаж устает;
- при высокой fatigue растёт burnout;
- событие пишется в log.

---

### 22.5. Release resolver

При релизе:

- считается release risk;
- выдаётся outcome;
- меняются trust/debt/value;
- карточка уходит в Done;
- при плохом outcome создаётся bug/incident follow-up;
- событие пишется в log.

---

### 22.6. WIP friction

Если active WIP выше лимита:

- команда получает дополнительную fatigue;
- debt растёт.

---

### 22.7. Sprint review

Каждые 5 дней:

- показывается summary;
- начисляется budget;
- игрок может выбрать межспринтовое действие.

## 24. Главный тест прототипа

После 5–10 партий ответить на вопросы:

```txt
1. Были ли ситуации, где релизить сейчас страшно, но ждать тоже страшно?
2. Были ли 3+ разумных хода в одном состоянии?
3. Были ли разные успешные стратегии?
4. Был ли смысл в найме разных ролей?
5. Стала ли усталость реальным фактором, а не декоративной шкалой?
6. Возникал ли WIP-хаос?
7. Было ли понятно, почему случился плохой релиз?
8. Хотелось ли сыграть ещё раз с другой командой?
```

Если большинство ответов “да” — ядро живое.

Если игрок всегда строит один маршрут:

```txt
analysis → dev → review → qa → release
```

и выигрывает — прототип провалил проверку.

## 26. Ограничения для разработчика

Разработчик не должен расширять объём без отдельного решения.

Запрещено в рамках первого прототипа:

- добавлять новые ресурсы без необходимости;
- делать сложную визуальную систему;
- добавлять narrative arcs;
- делать drag-and-drop, если кнопки быстрее;
- делать реальные технические решения карточек;
- добавлять карточки “архитектурных паттернов”;
- делать LLM-генерацию;
- делать систему департаментов;
- делать рынок сотрудников сложнее candidate pool;
- делать сложный календарь.

Главный фокус:

```txt
карточка → работа → усталость → риск → релиз → последствия
```

## 27. Ожидаемый результат

На выходе должен быть playable prototype, в котором за 15–25 минут можно пройти один run и понять:

- есть ли напряжение релиза;
- работают ли персонажи как “рука” игрока;
- создаёт ли усталость реальные компромиссы;
- создаёт ли WIP pressure игру;
- хочется ли пробовать другой билд команды.

Прототип считается успешным не тогда, когда он “весёлый и красивый”, а когда он честно отвечает:

> В этом пивоте есть игровое ядро или мы снова строим симулятор процесса без игры?
