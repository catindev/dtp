## Open Balance Hypotheses

### H1: QA Recheck Is Structurally Too Expensive

Evidence:

- `PAY-004` and `REP-011` failed on the same final recheck shape.
- The recheck is automatically created after post-QA dev/SRE work.
- It appears late, when deadlines and QA stamina are already under pressure.

Possible future knobs:

- make recheck `optional` instead of `important` in some cases;
- let recheck be shorter than full QA;
- let stale QA remain a risky release state without forcing a visible final task;
- reduce stamina drain for recheck;
- make recheck progress partially carry across exhaustion/day boundary.

### H2: First Quarter Goal Is Too Low After `DAYS_PER_QUARTER=5`

Evidence:

- Day 4 before release: `quarterValue=223/75`.
- No quarter review had fired yet.

Do not tune immediately from one session, but first-quarter target likely needs recalibration once weekly/monthly/quarterly goals replace the placeholder.

### H3: Budget Feels Binary Around Thresholds

Evidence:

- Budget fell to `2`.
- Multiple outsource attempts failed at `need 3/4/6`.

This may be good friction, but if it feels like dead UI rather than a strategic shortage, add clearer affordance or cheaper limited fallback.

### H4: Rewards Must Not Erase The QA/Analyst Scarcity Loop

Current evidence says the only reliably working pressure source is not queue order or raw task volume. It is the scarcity of one QA/analyst-shaped resource, especially this loop:

```txt
QA finds bugs -> rework appears -> QA recheck is needed -> the same QA is already exhausted
```

This was confirmed twice in one real session:

```txt
PAY-004 -> PAY-012
REP-011 -> REP-018
```

Any reward that can cheaply reset this shortage on demand risks deleting the strongest working moment in the game. The clearest example is the PAY-012 situation: Mira-QA was blocked, budget was low, and the task was objectively hard to save. A cheap "restore QA" button would have removed that decision entirely.

Design principle:

```txt
Rewards should make the team stronger around the shortage,
not cancel the shortage itself.
```

The player should still hit QA scarcity regularly. Rewards should create rare, expensive, or delayed ways to route around it, not make QA scarcity disappear as a default action.

#### Existing Good Pattern: `processBoost`

`processBoost` is close to the right reward shape:

- it increases work speed through `boostFactor = 1 + processBoost / 100`;
- it is capped at `25%`;
- it does not clear `exhaustedToday`;
- it does not restore stamina directly;
- a character at zero stamina remains blocked.

Important nuance: because faster work finishes sooner, `processBoost` indirectly improves stamina efficiency per completed subtask. That is acceptable at the current cap, but it should be treated as both a speed boost and a soft stamina-efficiency boost when balancing future rewards.

#### Reward Idea 1: Energy Drink As Debt, Not Reset

Budget can buy an immediate stamina bump for one character, but the stamina is borrowed, not gifted.

Possible shape:

```txt
pay budget now
gain +25..35 stamina now
next day: lower overnight recovery or lower stamina cap
one use per character per day
```

Narrative read:

```txt
worked on caffeine today -> woke up worse tomorrow
```

This is the best first candidate because it creates an emergency lever without removing long-term scarcity. It can save one important moment but makes tomorrow harder.

Open tuning question:

```txt
fixed bump vs scalable borrow
fixed next-day penalty vs penalty proportional to borrowed stamina
```

#### Reward Idea 2: Crisis Fund As Reactive Cleanup

The fund should only be buyable after a consequence has materialized, ideally on Morning Briefing.

Allowed use:

```txt
fallout exists now
pay budget
close or soften one specific fallout tail without team work
```

Avoid:

```txt
pre-buy insurance before dirty/risky releases
```

If the player can buy protection before risk, it becomes routine insurance and removes the intended release tradeoff. If the player buys cleanup after damage is visible, it becomes a meaningful emergency decision.

Guardrails:

- only on Morning Briefing;
- only after the fallout task exists;
- at most one per morning;
- expensive enough that a bad release day cannot pay for all of its own cleanup;
- should not bypass terminal chain limits for free.

#### Reward Idea 3: Role Slot Expansion

This should be structural, expensive, and rare:

```txt
temporary contractor for N days
or permanent secondary-specialty upgrade
or new hire / roster slot
```

This is the first real building block for the future deck/roster meta. It should require several good days or quarter rewards to afford. It should not be purchasable from one ordinary release.

Design position:

```txt
do not implement first
wait until the current QA scarcity is clearly readable to the player
then add role expansion as a long-term answer
```

#### Reward Idea 4: Outsource Capacity Upgrade

Permanent outsource discount or parallel outsource can help, but it is riskier because it can turn budget into a team replacement.

Keep outsource imperfect:

- lower quality than internal specialists;
- expensive for `critical`;
- less reliable for QA/SRE gates;
- should not fully clear `changed_after_qa` risk by itself.

This can be a late reward, not an early pressure valve.

#### Reward Acceptance Test

Before adding a reward, run this question against the known PAY-012 session:

```txt
If this reward existed, would the Mira-QA / PAY-012 moment disappear,
or would it remain but become slightly less punishing?
```

Interpretation:

```txt
disappears -> reject or make more expensive/rarer
remains but is less painful -> acceptable candidate
```

#### Harness Metrics To Track

After implementing any reward, the harness should check whether QA scarcity still appears under reasonable play:

- how often QA reaches low stamina;
- how often QA is needed but unavailable;
- how many tasks release risky because of QA/recheck pressure;
- how often the player buys emergency stamina specifically for QA;
- whether `changed_after_qa` and `Re-test changes after rework` still create meaningful decisions;
- whether clean play becomes dominant because rewards erase all late-stage risk.

The first structural check is binary:

```txt
Does QA scarcity still regularly happen after rewards exist?
```

If no, the reward is too strong. If yes, then exact prices can be tuned by playtest.
