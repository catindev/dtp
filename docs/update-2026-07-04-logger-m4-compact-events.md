# Logger Refactor M4: Compact Game Events

**Date:** 2026-07-04  
**Version:** 0.1.43  
**Save schema:** unchanged, `rt-campaign-v6`  
**Status:** implemented and verified with `npm run check` and `npm run debug`.

## Goal

Make gameplay logs analyzable as events, not as copied UI text or partial state dumps.

## Behavior Change

Game events now use `buildGameEventTelemetry`.

The payload keeps:

- `channel: "game_event"`;
- `eventType`;
- event time and current game day;
- `taskId`;
- `characterId`;
- `characterRole`;
- `subtaskRole`;
- `effects`;
- structured event `data`;
- compact resource values.

The payload no longer includes:

- `title`;
- `body`;
- full game resources object by reference;
- large morning-report arrays in action logs.

## Risk Analysis

Risk: losing human-readable event text can make raw JSON less convenient to skim.

Mitigation: the event type, ids, effects, and structured data remain. Human-readable views should be generated by analysis tools or UI, not stored as the primary source of truth.

Risk: some older engine events do not yet provide `data.taskId`.

Mitigation: telemetry temporarily derives `taskId` from the event title prefix when needed. This is a bridge toward adding structured data at the event source.

Risk: action logs can still contain non-normalized payloads.

Mitigation: the largest known action payloads were trimmed. A later milestone adds size guardrails to catch regressions.
